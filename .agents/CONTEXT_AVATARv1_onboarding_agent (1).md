# AVATARv1.0 — КОНТЕКСТ И ЛОГИКА АГЕНТА ОНБОРДИНГА

> Документ описывает полную логику работы агента онбординга:
> что собирает, как измеряет, с чем сравнивает, как синхронизирует.

---

## 1. ОБЩАЯ ИДЕЯ СИСТЕМЫ

Пользователь приходит в Telegram MINI App и проходит онбординг через живой разговор с агентом.
Агент не анкетирует — он слушает и анализирует речь.

По итогу разговора система:
- определяет доминирующие архетипы пользователя и сферы 
- определяет активные сферы жизни (где боль и блокировка)
- определяет уровень сознания по шкале Хокинса в каждой архетипе сферы
- проявляет релевантные карточки из векторной базы  (изначально JSON база со всеми архетипами  и сферами)
- сохраняет профиль пользователя для дальнейшей работы

```
Разговор → Анализ речи → Профиль → Карточки → Рост
```

---

## 2. ДВА ВХОДА В СИСТЕМУ

На старте пользователь выбирает один из двух путей:

### "Доверяю себе" — LIVE SCAN
Агент задаёт 7 вопросов в живом диалоге.
Анализирует текст ответов через векторную базу.
Определяет архетип, сферы и уровень Хокинса из паттернов речи.

### "Доверяю астрологии" — уже реализована

Оба пути приводят к одному результату — профилю пользователя и набору проявленных карточек.

---

## 3. СТРУКТУРА ДИАЛОГА — 7 ФАЗ (LIVE SCAN)

Каждая фаза — один вопрос. Агент не задаёт два вопроса подряд.
После каждого ответа — одна фраза отражения, затем следующий вопрос.

| Фаза | Вопрос агента | Что собираем |
|------|--------------|--------------|
| 0 | Приветствие, установка контакта | Базовый тонус, первая реакция |
| 1 | "Опиши своё состояние прямо сейчас — одним словом" | Доминирующая эмоция, уровень Хокинса |
| 2 | "Что сейчас занимает больше всего мыслей?" | Активные сферы, порядок упоминания |
| 3 | "Есть что-то что хотел изменить давно — но не сдвигается?" | Хроническая точка боли, сигнал архетипа |
| 4 | "Что уже пробовал с этим делать?" | Паттерн действий, уточнение уровня |
| 5 | "Как выглядела бы твоя жизнь через год — честно?" | Горизонт желания, ясность цели |
| 6 | "Когда что-то идёт не так — ты злишься, уходишь в себя или контролируешь?" | Реакция на стресс, тень архетипа |
| 7 | — | Проявление карточек по score |

Агент может проявить карточки уже после фаз 1–3 если пользователь отвечает коротко.
Минимум для проявления: уровень Хокинса + одна активная сфера + один сигнал архетипа.

## 5. С ЧЕМ СРАВНИВАЕМ — ВЕКТОРНАЯ БАЗА

База: Supabase + pgvector

Агент сравнивает текст пользователя с карточками архетипов через косинусное сходство векторов (embedding similarity).

```
Текст пользователя → OpenAI Embeddings → вектор запроса
                                                ↓
                              Supabase pgvector similarity search
                                                ↓
                              Ranked list карточек по score (0.0–1.0)
```

Текст пользователя сравнивается со всеми текстовыми полями блока vector каждой карточки — они конкатенируются в единый векторный контент при синхронизации.

### Пороги score

| Score | Действие агента |
|-------|----------------|
| выше 0.7 | Проявить карточку активно |
| 0.5–0.7 | Упомянуть как связанное |
| ниже 0.5 | Не показывать |

В тестовом режиме агент проявляет все карточки выше порога 0.5 без ограничения по количеству.

## 6. СТРУКТУРА КАРТОЧКИ — ЭТАЛОН

Каждая карточка делится на два блока: vector и not_vector.

---

### Блок VECTOR — идёт в векторную базу

Все поля этого блока конкатенируются и векторизуются.
По ним агент ищет совпадение с речью пользователя.

**Теневое проявление** — как архетип проявляет себя в этой сфере из тёмной стороны. Деструктивные паттерны, блокировки, избегание.

**Световое проявление** — как архетип проявляет себя в этой сфере из силы. Здоровые паттерны, реализация, рост.

**Общее описание** — суть отношений этого архетипа с данной сферой. Нейтральное, без тени и света.

**10 уровней по шкале Хокинса** — как именно это состояние выглядит на каждом уровне сознания в данной сфере для данного архетипа:

| # | Уровень | Название | Что описываем в карточке |
|---|---------|----------|--------------------------|
| 1 | 0–20 | Небытие / Стыд | Полный коллапс в сфере. Паралич воли. Нет контакта со сферой вообще |
| 2 | 20–100 | Горе / Вина | Скорбь об утраченном в сфере. Самообвинение. Ощущение что сфера потеряна навсегда |
| 3 | 100–175 | Страх / Тревога | Избегание сферы из страха. Тревожный контроль. Боязнь разоблачения |
| 4 | 175–200 | Гнев / Гордыня | Агрессия и манипуляция в сфере. Доминирование. Обвинение других |
| 5 | 200–250 | Мужество | Первая честность в сфере. Ответственность вместо обвинений. Начало реального действия |
| 6 | 250–310 | Нейтральность | Стабильное функционирование в сфере без драмы. Базовый порядок |
| 7 | 310–400 | Готовность | Активный рост в сфере. Видит возможности. Инициатива и открытость |
| 8 | 400–500 | Разум | Системное понимание сферы. Аналитика. Осознанное мастерство |
| 9 | 500–600 | Любовь / Радость | Действие в сфере из любви. Служение. Состояние потока |
| 10 | 600–1000 | Мир / Просветление | Растворение проблематики сферы. Сфера перестаёт быть задачей — становится выражением бытия |

**Маркеры речи теневого состояния** — конкретные фразы которые пользователь произносит когда находится в тени этого архетипа в данной сфере.

**Маркеры речи светлого состояния** — конкретные фразы которые пользователь произносит когда архетип проявлен в силе в данной сфере.

---

### Блок NOT_VECTOR — метаданные логики

Эти поля не векторизуются. Они управляют поведением системы после того как карточка найдена.

**Идентификатор карточки** — уникальный код карточки. Формат: СФЕРА_НОМЕРАРХЕТИПА

**Номер архетипа** — числовой идентификатор от 0 до 21

**Название архетипа** — текстовое название архетипа

**Сфера** — к какой сфере жизни относится карточка

**Диапазоны Хокинса** — текстовые метки трёх диапазонов (низкий / средний / высокий) для фильтрации при поиске

**Сигналы приоритета** — список сигналов в речи пользователя по которым агент выбирает именно эту карточку как первоочередную среди всех найденных

**Вес карточки** — числовой множитель важности. Влияет на итоговый score при ранжировании

**Минимальный порог** — ниже этого значения similarity score карточка не проявляется

**Квантовый сдвиг** — конкретное действие или осознание которое агент предлагает пользователю после проявления карточки. Следующий практический шаг.

**Вопрос агента** — вопрос который агент задаёт для активации и углубления работы с карточкой

**Связанные карточки** — список идентификаторов карточек которые открываются следующими. Формирует цепочку роста.

**Уровень прогрессии** — к какому уровню системы относится карточка

**Награда** — количество опыта которое пользователь получает за проработку карточки

## 7. ЛОГИКА ПРОЯВЛЕНИЯ КАРТОЧЕК

```
1. Агент получает все ответы пользователя из диалога
2. Конкатенирует текст в один запрос
3. Векторизует через OpenAI Embeddings
4. Запускает similarity search в Supabase pgvector
5. Получает ranked list карточек со score
6. Применяет вес карточки из not_vector
7. Фильтрует по минимальному порогу
8. Сортирует по итоговому score
9. Проявляет карточки по убыванию приоритета
```

Для каждой карточки агент:
- отражает что именно услышал в словах пользователя
- задаёт вопрос активации из карточки
- после ответа пользователя — показывает квантовый сдвиг
- начисляет награду за проработку
- предлагает связанные карточки как следующий шаг

## 8. ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ — ЧТО СОХРАНЯЕМ

После онбординга в базе фиксируется:
- идентификатор пользователя и его Telegram
- тип онбординга — LIVE SCAN 
- сырые ответы из диалога
- определённый доминантный архетип
- активные сферы в порядке приоритета
- уровень Хокинса
- список проявленных карточек с их score
- накопленный опыт и текущий уровень прогрессии

## 9. СИНХРОНИЗАЦИЯ ПОСЛЕ ОНБОРДИНГА

### Обновление профиля пользователя

После каждого последующего диалога агент:
- пересчитывает уровень Хокинса по новым ответам
- обновляет приоритеты активных сфер
- добавляет новые проявленные карточки
- начисляет опыт за завершённые карточки
- открывает связанные карточки если текущая проработана

### Синхронизация базы карточек

Карточки ведутся в JSON файлах и синхронизируются в Supabase по следующей схеме:

```
JSON файл обновлён или добавлен
        ↓
Скрипт синхронизации
        ↓
Читает блок vector → конкатенирует все текстовые поля → векторизует
        ↓
Обновляет векторное представление в Supabase
        ↓
Обновляет метаданные not_vector
```

Синхронизация запускается при:
- добавлении новых карточек
- обновлении текстов существующих карточек
- изменении маркеров речи или сигналов приоритета

## 10. ПРАВИЛА ВЕДЕНИЯ ДИАЛОГА ДЛЯ АГЕНТА

```
✓ Один вопрос за раз — никогда не два подряд
✓ После каждого ответа — одна фраза отражения
✓ Не называть уровень Хокинса или архетип вслух
✓ Не торопить — пауза и короткий ответ тоже данные
✓ Язык простой — никаких терминов системы в диалоге
✓ Агент слушает — пользователь чувствует что его слышат
✗ Не анкетировать — это живой разговор
✗ Не интерпретировать вслух — только отражать
✗ Не давить если пользователь уходит от темы
```
