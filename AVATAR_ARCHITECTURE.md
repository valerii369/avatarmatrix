# Архитектура проекта AVATAR

## 1. Введение
Этот документ представляет собой единственную точку истины (Single Source of Truth) по архитектуре системы AVATAR (версия 1.0). Система представляет собой Telegram Mini App (TMA), который использует искусственный интеллект, астрологические данные и механизмы работы с подсознанием для создания персонализированного цифрового портрета пользователя (Digital Soul Blueprint). Приложение имеет геймифицированную механику с рейтингами ("Хокинс"), энергией и карточками.

## 2. Технологический стек
- **Frontend:** Next.js 15, React 19, TypeScript, Tailwind CSS, Framer Motion, Zustand (управление состоянием).
- **Backend:** Python 3.11+, FastAPI, SQLAlchemy (Async), Uvicorn.
- **Инфраструктура/БД:** Supabase (PostgreSQL), `pgvector` (для RAG/семантического поиска), Vercel (развертывание Frontend), VPS Timeweb (развертывание Backend/Bot).
- **ИИ-провайдер:** OpenAI (модели GPT-4o, GPT-4o-mini, text-embedding-3-small/large).
- **Библиотеки ядра (Астрология):** `pyswisseph` (эфемериды Швейцарии), `geopy`, `timezonefinder`.
- **Интеграция Telegram:** `python-telegram-bot` (Telegram Bot API, TMA InitData validation).

## 4. База Данных (Supabase PostgreSQL + pgvector)
Основа приложения базируется на следующих ключевых таблицах:

1. **`users` (`User`)**: Идентификация Telegram (`tg_id`), данные о рождении, геймификация (`energy`, `xp`, `evolution_level`, `title`).
2. **Астрология & DSB**:
    - **`natal_charts` (`NatalChart`)**: Кэшированные астрологические данные по западной астрологии (планеты, куспиды, дома). 
    - **`dsb_digital_portraits`**: Выходы DSB пайплайна (факты, паттерны, рекомендации).
3. **Геймификация & Состояние**:
    - **`card_progress` (`CardProgress`)**: Таблица, отслеживающая статус 264 архетипов по 12 сферам (Locked, Recommended, Synced, Aligned).
    - **`sync_sessions` (`SyncSession`)**: Сессии текстовой синхронизации (5-шаговый pipeline агентов).
    - **`align_sessions` (`AlignSession`)**: WebSocket-сессии согласования и интеграции.
4. **Векторная база (RAG)**:
    - **`avatar_cards` (`AvatarCard`)**: Архетипы с внедрением поля `embedding` (`Vector(1536)`).
    - **`user_memory` (`UserMemory`)**: Векторизованные атомарные инсайты из чатов с Друг-ассистентом.

## 5. Основные Модули и Пайплайны (Backend)

### Digital Soul Blueprint (DSB)
Пайплайн, формирующий детальный слепок личности, разбитый на слои. Запускается при регистрации (онбординге). Оркестратор находится в `dsb/pipeline/orchestrator.py` и имеет следующую структуру:
1. **Layer 1 (Calculators)**: Математические расчеты по дате, времени и месту рождения (Western Astrology — реализована; Дизайн Человека, БаЦзы, Цолькин — в TODO-этапе 2/3).
2. **Layer 2 (Interpreters)**: Конвертация сырых данных в Universal Insight Schema (UIS) — единый стандарт фактов.
3. **Layer 3 (Synthesis)**:
    - **Merger**: Группировка по сферам.
    - **SphereAgent**: GPT-4o-mini формирует 5-слойный портрет (факты, цепи, паттерны, рекомендации, тень) индивидуально для каждой из 12 сфер.
    - **MetaAgent**: Выявление 3-7 "красных линий" (межсферных суперпаттернов).
    - **Compressor**: Генерация короткого брифа для отображения на Frontend-е.

### LLM Агенты (Cognitive Architecture)
- **Sync Agent (`agents/sync_agent.py`)**: Обрабатывает 5-шаговые текстовые "сеансы" с применением проективных сцен (нарративный сканер). Поддерживает соматические проверки с помощью LLM (проверка того, чувствует ли пользователь тело или абстрактно умничает).
- **Analytic Agent (`agents/analytic_agent.py`)**: Финальное Зеркало (Mirror). Берет полный транскрипт сессии и формирует вывод о Паттернах, Теневых качествах и уровне вибрации по шкале Хокинса.
- **Align Agent (`agents/align_agent.py`)**: Выполняется по WebSocket. Многошаговый диалог (вход, распаковка, интеграция, фиксация).
- **Assistant Agent ("Friend") (`agents/assistant_agent.py`)**: Чат со свободным общением. Использует RAG по прошлым сообщениям (`UserMemory`) и определяет резонанс по сферам.

## 6. RAG Система и Векторный Поиск
Реализовано через расширение `pgvector`.
- Генерация эмбеддингов идет с помощью `text-embedding-3-small` или `text-embedding-3-large`.
- Модуль `core/astrology/vector_matcher.py`: Осуществляет семантический сопоставительный поиск астрологических описаний текстов по отношению к архетипам (AvatarCards). Поддерживает мульти-векторный поиск (тень/свет/унификация).
- Модуль `dsb/storage/search.py`: Выполняет "cosine distance" по таблицам фактов и паттернов DSB для ответов на контекстные запросы пользователя в сессиях диалога.

## 7. Frontend Архитектура (Next.js Application)
- **Инициализация**: Telegram WebApp API передает `initData`, которая валидируется на бэкенде.
- **Store Management**: Реализован через Zustand (используются сторы `useUserStore` и `useCardsStore` в `lib/store.ts`).
- **Сетевой слой**: SWR используется поверх Axios-клиента (`lib/api.ts`) для кеширования и автоматической ревалидации профильного состояния.
- **Основные Компоненты**:
  - `MasterHubView.tsx`: Визуализация 12 сфер (Данные DSB Пайплайна).
  - `ConsciousnessVisualization.tsx`: Сложная визуализация карточек пользователя.
  - Навигация нижнего таббара с использованием Framer Motion.

## 8. Трейсинг Основных Потоков

### A. Поток Онбординга
1. **Frontend /onboarding**: Запрашивает дату, время и локацию рождения. Использует Google Places API для геокодинга.
2. **Backend POST `/api/auth/profile`**: Регистрация DTO.
3. **Task Orchestrator `initialize_onboarding_layer`**: 
    - Рассчитывает Натальную карту по Западной астрологии.
    - Инициализирует 264 пустые записи `CardProgress` (по 22 аркана в 12 сферах) для этого пользователя.
    - Асинхронно кидает задачу генерации полного DSB Portrait в `BackgroundTasks`.
4. Пользователь перенаправляется на главную страницу, где видит начальный XP и пустые карточки до момента завершения асинхронной обработки DSB.

### B. Поток Текстовой Синхронизации (SyncSession)
1. **Frontend**: Запрашивает `/api/sync/start` для конкретного `archetype_id`. Бэкенд вычитает энергию (-20).
2. **Backend**: Поднимает 5 сцен из таблиц. 
3. Пользователь общается с системой; Фронтенд шлет ответы на `/api/sync/phase`.
4. Бэкенд через `SyncAgent` считывает уровень ответа.
5. При достижении последней фазы вызывается `AnalyticAgent` (Mirror), который генерирует результаты. Добавляется XP (+50).
6. Карточка переходит в статус `SYNCED`. Состояние `SyncSession` маркируется `is_complete=True`.

## 9. Технический Долг и "Known Issues"
1. **Заглушки Калькуляторов**: В директории `dsb/calculators/*` системы Tzolkin, Бацзы, Генные Ключи, Human Design и Ведическая астрология содержат комментарии уровня "TODO: Phase 2/3 — implement". Пока работает только `western_astrology`.
2. **Восстановление DSB**: В роуте `dsb/api/routes/portraits.py` (line 449) API-эндпоинт `regenerate` является заглушкой: `message: "Перегенерация запущена (TODO: реализовать полный rerrun)"`.
3. **Хардкод Токеномики**: Файл конфигурации `config.py` содержит хардкод настроек энергии (например, `SYNC_ENERGY_COST = 20`).
4. **Обход БД При Разработке**: В методе запуска базы иногда стоит вычитка фантомных миграций для предотвращения зависаний (локальный `init_db`).

## 10. Рекомендации по Интеграции в Новый Проект
1. **Асинхронность**: Сохранить и перенести архитектуру асинхронного `BackgroundTasks` для оркестратора DSB. Это самая ресурсоемкая операция (осуществляет 14 вызовов к LLM последовательно).
2. **PostgreSQL & Vector**: PgVector глубоко интегрирован в SQL-выражения (`<=>` оператор для косинусного сходства). При переносе БД убедиться, что расширение Vector подключено до запуска Alembic.
3. **Обработка Telegram InitData**: Перенос авторизации обязателен. Функция проверки хеша Telegram (HmacSHA256) является единственным щитом безопасности всего API бэкенда.
